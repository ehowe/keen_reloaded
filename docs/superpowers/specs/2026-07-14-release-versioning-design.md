# Release Versioning — Design Spec

**Date:** 2026-07-14
**Status:** Approved
**Goal:** Full release pipeline — version tags, changelog generation, in-game version display.

## Context

keen_reloaded has no version tracking today:

- No git tags exist.
- No `VERSION` file.
- `project.godot` has no `application/config/version`.
- `export_presets.cfg` (gitignored, auto-generated by Makefile) hardcodes `short_version="1.0"` and `version="1.0"` for the macOS preset.

The Makefile already owns the build pipeline (`build`, `build-all`, `templates`, `test`, `run-app`). Commits follow conventional-commit format consistently (`feat()`, `fix()`, `docs()`, `chore()`, `test()`).

## Decisions

| Decision | Choice |
|----------|--------|
| Version scheme | **CalVer** `YYYY.MM.DD` (zero-padded). Same-day re-release → `.2`, `.3` suffix. |
| Release trigger | **Manual `make release`** — one command bumps, changelogs, commits, tags. |
| Changelog | **Auto-generated** from conventional commits since last tag. |
| Implementation | **Pure Makefile + POSIX shell** in `tools/`. Zero new deps. |

## Components

### 1. `VERSION` file (source of truth)

Plain text file at repo root, single line, no prefix:

```
2026.07.14
```

Git-tracked. Every release updates it.

### 2. `make release` target

Workflow (all steps automated, one command):

1. **Determine target version.** Read `VERSION`. If it already equals today's date, bump the trailing counter (`2026.07.14` → `2026.07.14.2` → `2026.07.14.3`). Otherwise set to today's date.
2. **Write `VERSION`** with the new version string.
3. **Patch `project.godot`** — set `application/config/version = "<version>"` via `sed`. If the key doesn't exist yet, insert it under `[application]`.
4. **Regenerate `export_presets.cfg`** — the existing Makefile generator already templates `short_version`/`version`; update it to read from `VERSION` instead of hardcoding `"1.0"`.
5. **Generate changelog** — `git log <last-tag>..HEAD`, group by conventional-commit type:
   - `feat` → **Added**
   - `fix` → **Fixed**
   - `perf` → **Performance**
   - `docs` → **Documentation**
   - `refactor` → **Refactored**
   - `test`/`chore`/`build`/`ci`/`style` → omitted (internal noise)
6. **Prepend section** to `CHANGELOG.md` with header `## v<version> — <date>` and grouped bullet list. Create file if absent.
7. **Stage + commit** — `git add VERSION CHANGELOG.md project.godot export_presets.cfg` → commit `chore(release): v<version>`.
8. **Tag** — `git tag v<version>`.
9. **Print guidance** — `"Release v<version> tagged. Push with: git push && git push --tags"`. Do **not** auto-push (irreversible without confirmation).

### 3. `tools/changelog.sh`

Pure POSIX shell script. Args: `<since-ref> <version>`.

- `since-ref` = previous tag, or empty (→ all history).
- Parses `git log <since-ref>..HEAD --format="%s"`.
- For each commit, extract type + scope + subject via conventional-commit regex.
- Output markdown grouped section to stdout.
- Exit nonzero if `git log` fails.

Mapping table (type → heading):

| Type | Heading |
|------|---------|
| `feat` | Added |
| `fix` | Fixed |
| `perf` | Performance |
| `docs` | Documentation |
| `refactor` | Refactored |
| `test`/`chore`/`build`/`ci`/`style` | (skipped) |
| Non-conventional (no type) | (skipped) |

Dedupe: if identical subject appears twice, keep first.

### 4. `make version` target (query, no side effects)

Prints:

```
VERSION:  2026.07.14
Last tag: v2026.07.14 (or: none)
Commits since tag: 5
```

Useful for sanity checks before release.

### 5. `make release-dry` target

Runs steps 1–2 (compute version, print what would change) **without** writing files, committing, or tagging. Lets you preview before committing.

### 6. In-game version display

`project.godot` gains `application/config/version` (set by `make release`).

Title screen reads it at runtime:

```gdscript
var _version: String = ProjectSettings.get_setting("application/config/version", "dev")
```

Displays in title screen corner as small text. If key absent (dev build, never released) → shows `"dev"`.

### 7. `export_presets.cfg` template update

The Makefile generator (lines 99–126 of `Makefile`) currently hardcodes:

```
'application/short_version="1.0"' 'application/version="1.0"' \
```

Update to read `$(shell cat VERSION)` (with fallback `"0.0"`) so exported builds carry the real version.

## Data Flow

```
VERSION file
   │
   ├──► make release  ──► patches project.godot
   │                     ──► regenerates export_presets.cfg
   │                     ──► tools/changelog.sh ──► CHANGELOG.md
   │                     ──► git commit + tag
   │
   ├──► make build   ──► export_presets.cfg uses VERSION for macOS version
   │
   └──► make version ──► prints status (read-only)
```

## Error Handling

- **`make release` when working tree dirty** → error out, refuse to release. Don't mix release commit with unrelated changes.
- **`tools/changelog.sh` with no commits since tag** → warn, still produce empty section with header only.
- **`make release` with no prior tag** → changelog covers full history.
- **`VERSION` file missing on `make build`** → fallback `"0.0"`, warn.

## Testing

- `make version` works on fresh clone (no tags yet).
- `make release-dry` prints correct target version without side effects.
- `tools/changelog.sh` correctly groups sample conventional commits.
- `make release` end-to-end: VERSION updated, `project.godot` patched, `CHANGELOG.md` created, tag created.
- `make build` still succeeds after VERSION changes.
- `./tests/run_all.sh` still passes (no game logic touched except optional title-screen label).

## Out of Scope

- CI/CD automation (GitHub Actions). Future work if desired.
- Binary artifact uploads to GitHub releases. Manual `make build-all` + attach for now.
- Pre-release / release-candidate channels. YAGNI until needed.
- Auto-push to remote. User controls push explicitly.
